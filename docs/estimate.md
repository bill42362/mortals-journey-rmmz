# 凡人修仙傳 RMMZ — 專案可行性評估報告

> 評估日期：2026-06-29
> 評估對象：[docs/requirements.md](./requirements.md) 所描述之專案
> 評估範圍：第一階段 Demo（序章青牛鎮 → 壓軸築基），以及長期願景的技術風險。

---

## 0. 結論摘要（TL;DR）

| 構面 | 評級 | 一句話結論 |
| --- | --- | --- |
| **第一階段 Demo 可行性** | 🟢 高 | 劇情線清楚、規模可控，用 MZ 完全做得出來。 |
| **完整願景可行性** | 🟡 中 | 13 套系統技術上多半可行，但**全部塞進 demo 不切實際**，必須做 MVP 切割。 |
| **「AI 做絕大部分」可行性** | 🟡 中偏可行 | 程式/文本/劇本 AI 是強項；**地圖與資料庫編輯是主要摩擦點**，需要建立工作流紀律。 |
| **美術素材** | 🟢 高 | 內建 + 已購買 12 組擴充包綽綽有餘。 |
| **核心精神還原** | 🟢 高 | 「苟/慢成長/小瓶/資源」四大精神**幾乎都是設計問題而非技術問題**，MZ 機制足以承載。 |
| **最大風險** | ⚠️ | ① 範圍蔓延（scope creep）② RMMZ 資料檔（JSON）的 AI 編輯與驗證流程 ③ 多套擴充包/外掛的相容性。 |

**總體判斷：值得做，且第一階段 demo 風險可控。** 成敗關鍵不在「能不能做」，而在「**敢不敢砍範圍**」與「**有沒有建立一條 AI 能安全產出並驗證 MZ 資料的工作流**」。

---

## 1. 評估方法與前提

- 本評估假設開發模式為：**1 名人類主導者 + AI（Claude Code）協作**，人類負責決策、地圖繪製與最終驗收，AI 負責程式、外掛、事件、文本。
- RPGMaker MZ 的專案本體是一組 **JSON 資料檔**（`data/*.json`）加上 **JavaScript 外掛**（`js/plugins/*.js`），以官方編輯器作為正規操作工具。
- 評估「可行性」分四級：🟢 容易 ／ 🟡 中等 ／ 🟠 困難 ／ 🔴 很難。

## 2. 引擎能力與「AI 編寫」可行性

### 2.1 RMMZ 本身的能力邊界

| 能力 | 說明 |
| --- | --- |
| 預設戰鬥 | 回合制（front/side view），可改逃跑率、先制/被偷襲、隊伍編成。原著「苟」的機制基礎充足。 |
| 事件系統 | 圖塊式地圖 + 事件指令；分歧、變數、開關、共通事件（common event）足以驅動大量敘事與系統邏輯。 |
| 外掛系統 | 純 JavaScript，可覆寫/擴充任意核心行為。**這是實現非典型系統（種植、煉丹、靈寵…）的主力。** |
| 資料庫 | actors / classes / skills / items / weapons / armors / enemies / troops / states / animations… 皆為 JSON，AI 可直接讀寫。 |

### 2.2 AI 能輕鬆做的部分（🟢）

- **外掛開發（JS）**：AI 的強項。種植、煉丹、典籍參悟、靈寵等自訂系統的程式碼。
- **劇本 / 對話文本 / 道具描述 / 技能說明**：純文字，AI 高度勝任。
- **資料庫批次內容**：技能、道具、敵人、狀態等 JSON 條目，可程式化大量產生。
- **事件腳本邏輯**：以共通事件 + 變數開關設計的系統流程。

### 2.3 AI 的主要摩擦點（🟡 / 🟠）

| 摩擦點 | 嚴重度 | 說明與緩解 |
| --- | --- | --- |
| **地圖圖塊繪製** | 🟠 | `Map###.json` 的 tile 圖層是巨大數字陣列，手工/AI 直接編寫極易出錯。**需求已正確預期此處保留人工**。緩解：地圖由人在編輯器畫，AI 只負責地圖上的「事件」。 |
| **地圖事件以 JSON 直編** | 🟡 | 事件指令的 JSON 格式冗長且易錯，且**編輯器開啟時會覆寫外部改動**。緩解：建立「誰是 source of truth」紀律——或全程在編輯器、或約定 AI 改完後人不在編輯器內動同檔。 |
| **無 headless 驗證** | 🟡 | MZ 沒有命令列建置/測試，AI 改完無法自動驗證，需人工啟動遊戲試玩。緩解：用 playtest、把可測邏輯抽到純 JS 外掛以利單元測試。 |
| **資料檔一致性** | 🟡 | 跨檔 ID 參照（技能↔職業↔敵人）若手改易斷鏈。緩解：寫產生器腳本維護 ID 對應，而非手改。 |

> **小結**：需求第 4 點「最小化人工參與，可能僅剩地圖繪製」**方向正確**，但實務上人類還需承擔：資料庫在編輯器內的最終確認、試玩驗收、外掛安裝設定。AI 無法 100% 無人化，但能扛下 70–85% 的產出量。

## 3. 各遊戲系統逐項可行性

> 評級同時標註「**Demo 必要性**」：◎ demo 核心、○ demo 可選、△ 建議延後至 demo 後。

| # | 系統 | 可行性 | Demo 必要性 | 實作路徑與備註 |
| --- | --- | :---: | :---: | --- |
| 1 | 神秘小綠瓶（加速生長） | 🟢 | ◎ | 變數計時 + 共通事件；與種植系統耦合。為核心循環樞紐，**必做且要先做**。 |
| 2 | 靈草種植 / 催化 | 🟢 | ◎ | 自訂外掛或事件化耕種；MZ farming 類外掛可參考。demo 主玩法。 |
| 3 | 技能 / 功法 | 🟢 | ◎ | 原生 skills/classes 系統，幾乎零風險。 |
| 4 | 典籍參悟 | 🟢 | ○ | 以時間/變數累積換取技能習得；外掛或事件皆可。 |
| 5 | 煉丹 | 🟢 | ◎ | 合成系統（配方 → 消耗素材 → 產出丹藥）；可用 crafting 類外掛。築基丹是壓軸關鍵。 |
| 6 | 製符 | 🟢 | ○ | 與煉丹同屬合成框架，複用即可。 |
| 7 | 煉器 | 🟢 | △ | 同合成框架，但 demo 非必要，建議延後。 |
| 8 | 法寶戰鬥（含自爆） | 🟡 | △ | 技能/道具可表現；「自爆」= 消耗法寶的一次性高傷技能，需自訂消耗邏輯與平衡。 |
| 9 | 靈寵培育 | 🟠 | △ | 需自訂養成系統（屬性成長、餵養、出戰），工作量大，**強烈建議延後**。 |
| 10 | 仙法戰鬥 | 🟢 | ○ | 原生技能系統可承載，後期再深化。 |
| 11 | 傀儡戰鬥 | 🟡 | △ | 召喚/臨時隊友機制，需外掛；demo 非必要。 |
| 12 | 符陣煉製 / 使用 | 🟠 | △ | 戰前布陣/場地效果偏自訂，複雜度高，延後。 |
| 13 | 屍傀戰鬥夥伴 | 🟡 | △ | 可轉化為「召喚出戰的隊友」，與傀儡共用框架。 |
| — | 殺人奪寶機制 | 🟡 | ○ | 設計面為主：可永久死亡的遭遇、屍體掉寶、NPC 互殺。**血色試煉是天然舞台**。需注意存檔/難度體驗。 |

**Demo 真正需要的核心子集**：①小綠瓶 ②靈草種植 ③技能功法 ④煉丹 ⑤基礎回合戰鬥 ⑥採藥/闖關 ⑦（輕量）殺人奪寶於血色試煉。其餘 9 項可在 demo 後逐步加入。

## 4. 美術與素材可行性 🟢

- 需求第 6 點「先以內建為主」+ 第 7 點 12 組擴充包，**素材量遠超 demo 所需**。
- 提醒：清單中部分品項並非 MZ 外掛/素材，而是**不同引擎或工具**——
  - `RPG MAKER MZ × ACTION GAME MAKER`：Action Game Maker 是**獨立引擎**，非 MZ 外掛，無法直接併入 MZ 專案（僅素材可能可轉用）。
  - `RPG Maker MZ × Action Combat Plugin`：這才是把 MZ 變成 ARPG 戰鬥的外掛，與原生回合制**二擇一**，需早期決策（見 §7）。
  - VX Ace / XP / MV bundle：主要價值在**素材轉用**，引擎本身不混用。
- ~~風險：跨包美術風格不一致~~ → **重新定位為設計資產**：凡界篇橫跨地域/大陸，環境風貌本就迥異，跨包混搭有助「一地一風貌」。策略改為**依地域指派風格家族**（如血色禁地用 Cthulhu Mythos 包營造詭譎感）；唯一原則為**同一場景/地域內保持一致**。詳見 [TECH §6](./TECH.md)。

## 5. 核心精神的可實現性 🟢

四大精神**絕大多數是設計問題，MZ 機制足以承載**：

| 精神 | 技術承載 | 風險 |
| --- | --- | --- |
| a. 苟是美德 | 逃跑率、先制/偷襲、可埋伏的事件編排、誤判懲罰；以掉落/狀態獎勵謹慎行為 | 低（設計工作為主） |
| b. 成長是熬出來的 | 自訂經驗/境界曲線、突破為事件門檻而非升級即達 | 低 |
| c. 小瓶是命脈 | 即系統 #1 + #2 的核心循環 | 低 |
| d. 資源即生命 | 經濟稀缺設計、丹藥/符/靈石為關鍵消耗品 | 中（數值平衡需反覆調校） |

> 真正的挑戰不在「能不能做出機制」，而在「**數值與節奏的反覆調校**」——這是耗時但低技術風險的工作。

## 6. 風險清單與緩解

| 風險 | 機率 | 衝擊 | 緩解策略 |
| --- | :---: | :---: | --- |
| **範圍蔓延**：13 系統全做進 demo | 高 | 高 | 嚴格 MVP 切割（§3、§7），延後 9 項系統。 |
| **MZ 資料檔編輯/覆寫衝突** | 中 | 高 | 訂定 source-of-truth 紀律；地圖只在編輯器畫；AI 改 JSON 時人不開同檔。 |
| **無自動驗證**：改了不知對不對 | 高 | 中 | 邏輯抽到純 JS 外掛做單元測試；建立 playtest 檢查清單。 |
| **外掛相容性**：多套包混用衝突 | 中 | 中 | 早期鎖定外掛清單；逐一安裝驗證；偏好同一生態系（如 VisuStella）。 |
| **戰鬥系統路線未定**（回合制 vs ARPG） | 中 | 高 | demo 前先決策（§7），避免重做。 |
| **數值平衡耗時** | 高 | 中 | 先求循環跑通，平衡放最後；用變數集中管理數值便於調整。 |
| **AI 無法真正 100% 無人化** | 高 | 低 | 接受人類保留地圖、驗收、外掛設定；以此設定期望。 |

## 7. 關鍵決策紀錄（已拍板 2026-06-29）

| # | 議題 | 決策 | 要點 |
| --- | --- | --- | --- |
| 7.0 | **Demo 範圍** | 只做 **7 項核心系統** | 小綠瓶、靈草種植、技能功法、煉丹、回合戰鬥、採藥/闖關、輕量殺人奪寶。其餘 9 項延後。 |
| 7.1 | **戰鬥系統路線** | **原生回合制** | 對「苟/逃跑/算計/資源」承載最直接，AI 與外掛生態最成熟。ARPG 留作後期實驗。 |
| 7.2 | **外掛生態主線** | **以 VisuStella MZ 為核心骨架** | 戰鬥、合成、選單、技能習得統一在 VisuStella 生態，降低相容性風險。 |
| 7.3 | **「境界」資料模型** | **職業=境界 混合模型** | 職業=境界（凡人/練氣/築基），等級=層，等級上限=撞瓶頸，突破=消耗築基丹的事件後 `Change Class`；功法用 VisuStella **Skill Learn System** 顯式習得、跨境界保留（即「典籍參悟」本體）。練氣 13 層用等級區間對應，**不開 13 個職業**。另設變數鏡像當前境界以利事件判斷。 |
| 7.4 | **死亡懲罰 / 殘酷度** | **標準 RPG + 根性留人 + 劇情之壁軟退場** | 死亡＝讀檔重來（利於測試與玩家反覆挑戰），殘酷只存在於**敘事**。允許玩家以根性反覆嘗試，**讓某些原著中註定殞落的角色活下來**，圓滿讀者/觀眾心中的遺憾；但當真正撞上**劇情之壁**（canon 不可違逆處）時，改用**機制軟退場**（非血腥死亡，優雅地讓角色離場）以尊重原著走向。不做永久死亡。 |
| 7.5 | **AI–編輯器工作流** | **混合分權** | ① 地圖 tile 圖層永遠人工繪製，AI 絕不碰陣列。② 事件邏輯集中到 **Common Event + 外掛**，地圖事件只留薄殼（呼叫共通事件），由 AI 產出。③ 資料庫大量內容由 AI 以**產生器腳本**輸出合法 JSON。④ `js/plugins/` 與參數 AI 全權、git 追蹤。⑤ **鐵則：編輯器與 AI 不同時寫同檔——編輯器關閉後 AI 才動 `data/`**，並對 JSON 做格式正規化以利 git diff。 |

> **7.4 設計註記**：此取向把本作定位為「比原著仁慈、但敬重原著」的二創——預設體驗親民可重試，獎勵玩家的執著（圓滿遺憾），同時用「軟退場」守住 canon 的關鍵節點，避免硬改原著造成的違和。最殘酷的張力改由**敘事與資源稀缺**承載，而非永久死亡的數值懲罰。

## 8. 工作量與時程粗估

> 以 1 人 + AI 協作、非全職投入為前提，**僅供量級參考**，非承諾。

| 階段 | 內容 | 粗估量級 |
| --- | --- | --- |
| 文件 / 架構 | SPEC、TECH、ROADMAP、story、決策拍板 | 1–2 週 |
| 技術骨架 | 外掛選定、境界模型、小綠瓶+種植+煉丹核心循環 | 3–5 週 |
| 內容生產 | 序章→第三幕地圖事件、文本、戰鬥配置 | 4–8 週 |
| 壓軸與整合 | 血色試煉、築基、殺人奪寶、串接 | 2–4 週 |
| 調校 / 驗收 | 數值平衡、試玩、修 bug | 2–4 週（持續） |
| **合計（Demo）** | | **約 3–6 個月（彈性大）** |

最大變數仍是**範圍**與**數值調校的反覆次數**。

## 9. 總體建議

1. **做，且先做 demo 核心循環**：小綠瓶 → 種植 → 採藥 → 煉丹 → 變強，這條線跑通，專案就活了。
2. **狠下心砍範圍**：靈寵、傀儡、屍傀、符陣、煉器、法寶自爆等 9 項**全部延後**到 demo 之後。
3. **先拍板 §7 的五個決策**，尤其是戰鬥路線與境界資料模型，避免返工。
4. **建立 AI–MZ 工作流紀律**：地圖人工繪製、事件/資料/外掛由 AI 產出、約定 source of truth、建立 playtest 驗收清單。
5. **把數值集中、留到最後調**：先求「能玩」，再求「好玩」。

> 一句話：**技術不是瓶頸，紀律與取捨才是。** 第一階段 demo 完全值得推進。
